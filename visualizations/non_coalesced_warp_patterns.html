Warp Memory Access Patterns

Example: Warp 0 (Threads 0-31) in Block(0,0)
Threads 0-15 (threadIdx.y=0):
Calculation:
blockIdx.x=0, blockIdx.y=0, threadIdx.x=0-15, threadIdx.y=0
row = 0*16 + 0 = 0
col = 0*16 + (0-15) = 0-15
Memory index = row × 1024 + col = 0-15
Memory Indices:
Threads 16-31 (threadIdx.y=1):
Calculation:
blockIdx.x=0, blockIdx.y=0, threadIdx.x=0-15, threadIdx.y=1
row = 0*16 + 1 = 1
col = 0*16 + (0-15) = 0-15
Memory index = row × 1024 + col = 1024-1039
Memory Indices:
❌ GAP: 1009 elements
Between index 15 and 1024 = 1009 elements!
Example: Warp 1 (Threads 32-63) in Block(0,0)
Threads 32-47 (threadIdx.y=2):
Calculation:
blockIdx.x=0, blockIdx.y=0, threadIdx.x=0-15, threadIdx.y=2
row = 0*16 + 2 = 2
col = 0*16 + (0-15) = 0-15
Memory index = row × 1024 + col = 2048-2063
Memory Indices:
Threads 48-63 (threadIdx.y=3):
Calculation:
blockIdx.x=0, blockIdx.y=0, threadIdx.x=0-15, threadIdx.y=3
row = 0*16 + 3 = 3
col = 0*16 + (0-15) = 0-15
Memory index = row × 1024 + col = 3072-3087
Memory Indices:
❌ GAP: 1009 elements
Between index 2063 and 3072 = 1009 elements!
Problems
  • Gap between memory accesses = 1009 elements (or 1024 when counting the full stride)
  • GPU cannot coalesce these into a single memory transaction
  • Requires multiple memory transactions per warp
  • Pattern continues for all 8 warps in the block
✓ Solution: Use blockDim(32, 32) or (32, 16)
With blockDim.x = 32, all 32 threads of a warp stay within the same matrix row. They access consecutive memory locations (e.g., indices 0-31, 32-63, etc.) that can be coalesced into 1-2 transactions. This eliminates the 1009-element gaps and maximizes memory bandwidth utilization!
